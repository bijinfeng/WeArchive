## Problem Statement

设计一个微信聊天记录备份、管理和查看工具。第一版不以 AI 为主入口，而是先做可靠的本地档案核心：导入已有微信备份或导出数据，标准化保存，离线查看，搜索，导出，并为 V1.5 增量备份、未来 NAS/Web、未来 AI 检索和个人知识库打地基。

用户明确希望参考 weibeifen/wxbackup 的备份功能和聊天记录查看功能。wxbackup 的核心价值包括多微信账号、全量/增量备份、离线查看、联系人/群聊/公众号、聊天详情、多消息类型搜索、导出、访问密码、恢复能力。

## What Makes This Cool

最酷的不是“又一个聊天记录导出器”，而是把微信聊天记录变成一个可长期保存、可迁移、可查看、可搜索、未来可被 AI 引用的个人档案库。

V1 的 wow moment 是：用户导入一份微信数据后，马上看到一个像微信但更适合归档的查看器：多账号、会话、群聊、公众号、消息类型、附件、搜索、导出都在本地可用。未来 AI 回答任何问题时，都能点回原始聊天和附件，而不是生成不可验证的总结。

## Constraints

- V1 交付形态是个人电脑本地工具，不先做 NAS/Web 部署。
- 长期架构要支持未来包装成 NAS/Web 服务。
- V1 不复刻完整手机到设备的备份协议链路。
- V1 支持导入和管理已有微信备份或导出数据。
- V1.5 再补增量备份链路。
- AI 不作为 V1 主入口，但 V1 数据结构必须保留原文证据链。
- 参考 wxbackup 的信息架构，但不要照抄视觉或商业模式。
- 隐私优先：默认本地存储，本地索引，本地访问密码或加密。

## Premises

1. V1 的核心价值是导入已有微信备份/导出数据后，能可靠查看和搜索。
2. 产品核心应该是一个标准化本地档案格式，供桌面、NAS/Web、AI 共用。
3. 查看器要优先参考 wxbackup 的信息架构：多账号、会话列表、联系人/群聊/公众号、聊天详情、搜索筛选、消息类型渲染、导出、访问密码。
4. V1.5 再做增量备份链路，V1 数据结构从第一天给增量留字段。
5. AI 不做 V1 主入口，但必须为未来 AI 留证据链：原始消息、附件、时间、联系人、导入源、哈希或稳定 ID。

## Landscape Notes

- wxbackup 证明了用户需要的不只是“备份到设备”，还包括脱离微信后的离线查看、搜索、统计、导出和恢复。
- 微信官方备份链路的空缺是：备份文件通常不能直接在电脑上打开查看，只能用于恢复。这个缺口就是本产品 V1 的价值。
- 现有开源方向大多是 PC 数据库导出、iOS/macOS 数据导出、HTML/CSV 查看器。常见问题是微信版本兼容、密钥获取、消息类型不全、大数据量性能和附件缺失。

EUREKA: 官方备份解决“别丢”，但没有解决“备完以后能读、能搜、能长期管理”。V1 应该先占住这个缺口，而不是第一天就挑战完整备份协议。

## Approaches Considered

### Approach A: 最小可用导入查看器

支持导入一种或少数几种已有微信数据格式，转换成本地档案库，提供账号、联系人、会话、聊天详情、基础搜索和导出。

Effort: S/M
Risk: Low

Pros:
- 最快可用，能先验证查看器体验。
- 不被微信备份协议、端口、平台差异拖住。
- 适合先做一个可以展示、可以自己使用的版本。

Cons:
- 备份感较弱，用户需要自己先拿到数据源。
- 后续支持更多数据源时可能要改导入模型。
- 如果档案格式一开始设计不够好，AI 和 NAS/Web 会被拖累。

Reuses:
- wxbackup 的查看器信息架构。
- 现有导出工具的 HTML/CSV/SQLite 导入思路。

### Approach B: 本地档案核心 + 多导入器

先设计标准档案格式，再做多个导入器：wxbackup 风格备份目录、PC 微信导出数据、未来手机备份包。查看器和搜索只读标准档案。

Effort: M/L
Risk: Medium

Pros:
- 长期最稳，V1、V1.5、NAS/Web、AI 都复用同一个核心。
- 不被某一种数据源锁死，导入器可以逐步增加。
- 可以从第一天处理导入失败、附件缺失、消息类型未知、重复消息等真实问题。

Cons:
- 第一版设计工作更重，需要先定义档案格式和导入状态模型。
- 用户感知到的功能可能比底层工作少，需要用导入健康度和查看器体验体现价值。
- 多导入器容易膨胀，V1 必须限定 1-2 个优先数据源。

Reuses:
- wxbackup 的功能边界。
- 微信数据导出工具的导入经验。
- 本地搜索和附件归档的通用架构。

### Approach C: 备份优先的 wxbackup-like 产品

第一版就追求手机到本地设备备份，尽量复刻 wxbackup 的备份/恢复体验，同时带查看器。

Effort: XL
Risk: High

Pros:
- 产品定义最完整，最像真正的微信备份工具。
- 如果做成，用户上手路径最直接。
- 更容易讲清楚“我就是微信备份工具”。

Cons:
- 微信版本、端口、协议、系统差异会吞掉大量第一版时间。
- 查看器、搜索、导出可能被备份链路拖慢。
- 失败路径多，早期产品可信度风险高。

Reuses:
- wxbackup 的完整产品路径和流程文案。

## Recommended Approach

选择 Approach B：本地档案核心 + 多导入器。

理由：用户想要的是第一版可用的微信备份和查看工具，但长期又要 AI、个人知识库、高级检索和 NAS/Web。B 把最重要的地基先打对：标准档案格式、导入器、查看器、搜索、导出、证据链。V1 可用，V1.5 能补增量备份，未来 AI 能基于原文引用工作。

## V1 Scope Decision

V1 should ship one complete import path instead of many half-working adapters.

Recommended first import source:
- Primary: a normalized SQLite/CSV/HTML export produced by an existing local WeChat export tool or by a small fixture generated for this project.
- Secondary, only if a real sample is available: wxbackup-style backup directory import.

Why not start with wxbackup directory as the only source: the public wxbackup repo documents product behavior and UI well, but it does not expose a stable public backup schema. Treating it as the first and only import contract would make implementation depend on reverse engineering a moving target. The product should reference wxbackup's UX and feature boundary, while owning its own Archive v0 format.

V1.5 can add acquisition: incremental backup/import into the same Archive v0 model.

## Security Posture

V1 default:
- All data stays local.
- The archive is stored in a user-selected directory.
- The app has an access password/app lock for casual protection.
- Exported evidence packages are explicit user actions, never background sync.
- The app records file hashes for attachments and exported bundles.

V1.1 target:
- Optional archive encryption at rest.
- Encrypted search index or rebuildable search index, depending on performance.
- Per-archive recovery key export.

Reasoning: V1 must not pretend an app password is full encryption. For the first version, be honest: local-only plus app lock protects against casual access, while OS disk encryption protects the machine. Full archive encryption is valuable but should not block the first usable viewer.

## V1 Product Shape

### Primary Flow

1. 创建本地档案库。
2. 导入一个已有微信备份目录、导出包或 SQLite/CSV/HTML 数据源。
3. 导入器解析账号、联系人、群聊、公众号、会话、消息、附件。
4. 生成标准档案库和搜索索引。
5. 用户在本地查看器里按账号、会话、联系人、群聊、公众号浏览。
6. 用户按关键词、日期、消息类型、发送人、附件状态搜索。
7. 用户导出单个会话、搜索结果或证据包。

### Screens

- 档案首页：账号卡片、导入状态、最近备份/导入时间、档案大小、健康度。
- 导入向导：选择数据源、检测格式、预估内容、冲突策略、开始导入、导入报告。
- 聊天查看器：左侧会话/联系人列表，中间聊天流，右侧筛选/元数据/导出面板。
- 搜索页：关键词、日期范围、消息类型、联系人/群聊、附件状态。
- 导出页：HTML、CSV、JSON 清单、附件目录、可选哈希清单。
- 设置页：档案路径、访问密码、索引重建、数据健康检查、隐私选项。

### Message Types For V1

Must support:
- 文本
- 图片
- 视频
- 文件
- 语音占位和元数据
- 链接
- 小程序
- 位置
- 转账/红包占位和元数据
- 系统消息
- 名片
- 群公告
- 合并消息占位
- 未知消息类型占位

V1 不要求完美播放或还原所有类型，但必须做到：不丢记录、能显示占位、能保留原始 payload、能在导入报告里说明缺失或未知。

### Archive Data Model

Core entities:
- Archive: 本地档案库，包含版本、创建时间、加密/访问策略。
- Account: 微信账号，包含 wxid、昵称、头像、导入源。
- Contact: 好友、群聊、公众号、企业微信联系人。
- Conversation: 单聊、群聊、公众号线程。
- Message: 稳定 ID、会话 ID、发送人、时间、类型、文本摘要、原始 payload、导入源、哈希。
- Attachment: 文件路径、原始文件名、MIME、大小、哈希、可用状态、缩略图。
- ImportJob: 数据源、导入时间、状态、错误、重复数、缺失附件数。
- SearchIndex: 文本索引、类型索引、日期索引、联系人索引。

Key rule: 查看器和未来 AI 不直接依赖某一种微信导出格式，只依赖标准档案模型。

## Open Questions

- 桌面技术栈是 Electron/Tauri，还是先做本地 Web App？
- 语音消息 V1 是只展示占位，还是尝试转码播放？
- 导出证据包是否需要从 V1 就带哈希清单？我建议带，成本低，未来取证/AI 都用得上。

## Success Criteria

- 用户可以创建一个本地档案库并导入至少一种真实微信数据源。
- 导入完成后能看到账号、联系人/群聊/公众号、会话列表和聊天详情。
- 至少支持文本、图片、文件、链接、系统消息的可读展示。
- 搜索支持关键词、会话、日期范围、消息类型。
- 导出支持单会话 HTML 和搜索结果 CSV/JSON。
- 导入报告明确显示总消息数、成功数、跳过数、重复数、未知类型、缺失附件。
- 所有消息保留原始来源字段，为未来 AI 引用做准备。
- 大会话浏览不卡死，消息列表使用分页或虚拟滚动。

## Distribution Plan

V1:
- 桌面应用发布，优先 macOS/Windows。
- GitHub Releases 提供安装包。
- 本地档案库存放在用户选择的目录。
- 自动更新可以推迟到 V1.1。

CI/CD:
- GitHub Actions 构建 macOS/Windows 安装包。
- 每个 release 附带 checksum。
- 示例数据集和导入器回归测试随 CI 运行。

Future:
- NAS/Web 包装为同一核心的服务模式。
- V1.5 增量备份链路复用 ImportJob 和 Archive data model。
- AI 检索层只访问标准档案和引用索引。
